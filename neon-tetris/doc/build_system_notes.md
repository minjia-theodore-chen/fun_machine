# CMake & C++ Build System Learning Notes

This document summarizes our architecture design, file isolation logic, and the mechanics of the CMake build system in the `neon-tetris` project.

---

## 1. Directory Structure & File Isolation Logic

A professional C++ project maintains a strict boundary between public interfaces (declarations) and private implementations (definitions).

```
neon-tetris/
├── CMakeLists.txt         # Root build configuration
├── .clangd                # Editor configuration helper
├── .gitignore             # Git ignore patterns
├── include/               # Header Files (.hpp) - Declarations
│   └── Game.hpp
└── src/                   # Source Files (.cpp) - Implementation details
    ├── Main.cpp
    └── Game.cpp
```

### Why isolate headers in `include/`?
1. **Encapsulation**: Other parts of a system only need to look at the `.hpp` files to know *how* to interact with a module (functions and classes available), hiding implementation complexity inside `.cpp` files.
2. **Compilation Speed**: Header guards (`#pragma once`) prevent multiple inclusions. Isolating headers allows the compiler to resolve include search paths separately and prevents namespace pollution.
3. **Linker boundary**: Header files do *not* generate object code; they are included by `.cpp` files. `.cpp` files are compiled into separate binary `.o` object files, which are later sewn together by the linker.

## 2. CMake Build System Mechanics

`CMakeLists.txt` is not a script that compiles code; it is a **meta-build system generator** that creates standard Makefiles or IDE projects (like Xcode/Visual Studio) dynamically based on the platform.

### Core CMake Commands Used:
* `cmake_minimum_required(VERSION 3.12)`: Ensures the user has a modern CMake toolchain installed to avoid syntax errors.
* `project(neon-tetris LANGUAGES CXX)`: Declares the project name and sets C++ (CXX) as the required compiler toolchain.
* `set(CMAKE_CXX_STANDARD 17)`: Enforces modern **C++17** standards.
* `add_compile_options(-Wall -Wextra -Wpedantic -Werror)`: Tells the compiler to enable strict diagnostics and treat all warnings as compile errors (ensuring code quality).
* `set(CMAKE_EXPORT_COMPILE_COMMANDS ON)`: Tells CMake to output a JSON compilation database mapping source files to their compilation flags.
* `add_executable(neon-tetris src/Main.cpp ...)`: Tells CMake to build a final binary called `neon-tetris` using the specified source files.
* `target_include_directories(neon-tetris PRIVATE include)`: Specifies that the `include/` directory should be searched for header files during compilation. `PRIVATE` means this search path is only active when compiling `neon-tetris` itself.

## 3. Out-of-Source Build (源外构建)

A key concept in modern software engineering is separating the **Source Tree** (your pristine code) from the **Binary Tree** (temporary build artifacts).

### The Directory Walkthrough:
When you are inside the `build/` directory and run:
```bash
cmake ..
```
* **Command Location**: You remain inside `build/`.
* **Path Parameter (`..`)**: CMake looks at the parent directory (`..`) to find `CMakeLists.txt`. It evaluates all paths written inside `CMakeLists.txt` (like `src/Main.cpp`) relative to that parent directory.
* **Output Placement**: Because your shell is currently located inside `build/`, CMake writes all generated build cache files, compiler Makefiles, and compiled `.o` binaries **directly into your current `build/` directory**.

### The Clean Advantage:
To clean the project fully, you do not need `make clean` scripts. You simply delete the `build/` folder:
```bash
rm -rf build
```
This guarantees your source files remain 100% clean and free of binary garbage.

## 4. IDE Language Server Integration (IntelliSense)

For the IDE to show auto-completion, diagnostics, and code definitions, it runs a background language server (like `clangd` or Microsoft IntelliSense) that behaves exactly like the C++ compiler.

1. **The Compilation Database (`compile_commands.json`)**:
   - Generated by CMake inside `build/` during `cmake ..`.
   - Lists the exact commands used to compile each file (including the crucial `-Iinclude` compiler flags).
   - *Note: According to Clang specifications, this file contains local absolute directories, meaning it must **never** be committed to Git. We ignore it in `.gitignore`.*
2. **The Clangd Connector (`.clangd`)**:
   - Placed in the project subdirectory [neon-tetris/.clangd](file:///Users/chenminjia/fun_machine/neon-tetris/.clangd).
   - Contains a single relative instruction: `CompilationDatabase: "build"`.
   - This tells the language server to look inside the `build/` folder to find `compile_commands.json`. It bridges the gap between the clean root repository and the generated build artifacts.
